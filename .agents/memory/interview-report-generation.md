name: Interview Ace report generation
description: How the end-of-interview feedback report is generated robustly, why it's split into two AI calls, and why the final scores are computed deterministically (not taken from the AI).
---

## Split the report into two AI calls
The final report is generated by TWO separate AI calls, NOT one:
1. Main call → per-competency 1–5 ratings (covered competencies only), roleFit, and strengths / concerns / nextSteps. NO per-question array in this call.
2. A second, focused call → the per-question feedback array only.

**Why:** when everything was one call, a long per-question array pushed the JSON past the token cap and it truncated; parsing failed and the UI fell back to uniform ~60% scores (a real user-visible bug — "every score is 60%"). Keeping the main JSON small guarantees the ratings survive, and the per-question feedback rides in its own call that can't corrupt them.

**How to apply:** never let the scored report and the (unbounded) per-question list share one generation call. Parse defensively: strip ```json fences AND slice the outermost `{...}` (object) / `[...]` (array) before `JSON.parse`. Clamp/normalize every numeric field with a sane default; on total parse failure fall back to a neutral report rather than crashing.

## Final scores are computed DETERMINISTICALLY — the AI never sets the total
The AI supplies ONLY the per-competency 1–5 ratings (+ comments) and the prose. The Total Weighted Score, the 0–100 overall, and the hiring recommendation band are all computed client-side from those ratings (`deriveScores` → `weightedScoreFor(ratings, durationMin)`; overall = weighted × 20; band from `recommendationForWeighted`).

**Why:** an AI-chosen total can contradict its own per-competency ratings (and drift from the verdict/recommendation the candidate sees). Deriving everything from the ratings makes the scorecard internally consistent by construction. `weightedScoreFor` renormalises weights over ONLY the competencies the interview length covered, so a 10-min interview isn't penalised for the four competencies it never tested.

**How to apply:** keep score/overall/recommendation derivation OUT of the prompt and OUT of trusted-AI-output — compute them from the parsed ratings. Never add an AI-provided "overallScore"/"weightedScore"/"recommendation" back into the trusted path.

## Competencies live in feedbackJson (no schema change) + back-compat
The six-competency block is stored INSIDE the report's `feedbackJson` blob (`competencies?: Partial<Record<CompetencyKey, CompetencyRating>>`), not as new typed DB columns. The typed columns stay overall / communication / grammar / confidence / technical (overall = weighted × 20 keeps analytics/trends working).

**Why:** a rich, evolving structured block shouldn't require a migration; the canonical typed scores stay stable while the detailed scorecard stays free-form in the JSON blob.

**Back-compat (any consumer must tolerate BOTH shapes):** OLD saved reports used 0–100 competency `score` objects and an `improvements` array; NEW reports use 1–5 `rating` objects and a `concerns` array. Parse `concerns` with an `improvements` fallback, guard `competencies` for `undefined`, and derive the recommendation from the stored `overallScore` for old rows that have no ratings.
