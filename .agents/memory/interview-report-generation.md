## Split the report into two AI calls
The final report is generated by TWO separate AI calls, NOT one:
1. Main call → overall + per-skill scores, roleFit, strengths / improvements / nextSteps, and a competency-framework object. NO per-question array in this call.
2. A second, focused call → the per-question feedback array only.

**Why:** when everything was one call, a long per-question array pushed the JSON past the token cap and it truncated; parsing failed and the UI fell back to uniform ~60% scores (a real user-visible bug — "every score is 60%"). Keeping the main JSON small guarantees the scores/competencies survive, and the per-question feedback rides in its own call that can't corrupt them.

**How to apply:** never let the scored report and the (unbounded) per-question list share one generation call. Parse defensively: strip ```json fences AND slice the outermost `{...}` (object) / `[...]` (array) before `JSON.parse`. Clamp/normalize every numeric field with a sane default.

## Competency framework lives in feedbackJson (no schema change)
The HR competency assessment (functional knowledge, communication, collaboration, personality, education, IT skills, adaptability — score + comment each) is stored INSIDE the report's `feedbackJson` blob, not as new typed DB columns. The typed columns stay overall / communication / grammar / confidence / technical.

**Why:** a rich, evolving structured block shouldn't require a migration; the canonical typed scores stay stable for analytics/trends while the detailed framework stays free-form in the JSON blob. Any UI reading it must guard for `undefined` (older reports predate the field).
